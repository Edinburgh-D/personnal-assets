# 通用技能向不同业务场景的适配指南

> 原始技能骨架：情报搜集 → 内容/功能体系 → 产品方案 → 技术架构 → 代码Prompt库 → 设计原型 → 法律合规 → 测试验证
> 适用原则：骨架保留，根据业务类型替换内部填充

---

## 一、先判断业务类型

| 业务类型 | 核心交互 | 是否适用测评框架 | 适配策略 |
|----------|---------|----------------|---------|
| 性格/风格测评 | 答题→画像 | ✅ 直接套用 | 六阶段全适用 |
| 教培工具（消课/排课） | 数据录入→查询→统计 | ❌ 不适用测评框架 | 保留骨架，功能体系改为B端模块 |
| O2O线下服务（约球/预约） | 位置→预约→履约→评价 | ❌ 不适用测评框架 | 保留骨架，功能体系改为服务链路 |
| 内容社区 | 浏览→互动→创作 | ❌ 不适用测评框架 | 保留骨架，功能体系改为内容生产链路 |

---

## 二、通用技能中"永远不变"的部分

以下Phase无论做什么类型的小程序，流程和产出物都一样：

### P1 情报搜集（100%复用）
- 竞品调研 → 找同类教培工具/约球App，看功能清单
- 场景内容库 → 收集教培校长/台球爱好者的真实痛点
- 技术栈方案 → 还是Taro+Next.js那套
- 内容冷启动 → 教培行业金句/台球圈钩子文案
- 法律合规 → 教育行业特殊合规（如下）
- 裂变增长 → 教培转介绍机制/球房会员裂变

### P4 技术架构（80%复用）
- 前端框架还是 Taro/React/TypeScript
- 数据库设计思路不变（表结构随业务变）
- MVP裁剪原则不变

### P7 法律合规（需叠加行业特殊项）
- 教培行业需额外关注：
  - 教育部预付费资金监管要求
  - 不得一次性收取超3个月/60课时费用
  - 办学许可证相关（如做学科类）
- O2O线下服务需额外关注：
  - 用户人身安全免责声明
  - 场地责任划分

### GitHub协作流程（100%复用）

## 三、场景一：教培消课小程序（店长端+用户端）

### 业务本质
B端工具 + C端查询。核心是"课时包的增删改查"和"消课记录的可追溯"。

### 不适用测评框架的部分
- 不需要"题库"和"画像体系"
- 不需要"答题页"和"结果页"
- 不需要"海报裂变"（但可以有"课时耗尽提醒"作为分享触发点）

### 适配后的Phase体系

#### P2 功能体系（替代原"内容体系+画像体系"）

**双端功能拆解：**

**店长端（管理端）：**
| 模块 | 核心功能 |
|------|---------|
| 学员管理 | 新增学员、编辑档案、绑定家长微信 |
| 课程包管理 | 创建课程包、设定总课时、有效期、单价 |
| 消课操作 | 签到消课、批量消课、撤销纠错 |
| 数据统计 | 剩余课时汇总、到期预警、续费率 |
| 财务报表 | 收入统计、课时消耗趋势 |

**用户端（家长端）：**
| 模块 | 核心功能 |
|------|---------|
| 课时查询 | 总课时、已消、剩余、有效期 |
| 消课记录 | 时间线形式，每次消课详情 |
| 到期提醒 | 课时不足/即将到期推送 |
| 一键续费 | 跳转支付（企业主体）或联系店长（个人主体） |

**P2 产出物：**
- `店长端功能清单.md`
- `用户端功能清单.md`
- `数据字典.md`（学员/课程包/消课记录/门店表结构）

#### P3 产品方案（重写）

**核心流程：**
```
店长端：登录 → 选择门店 → 学员列表 → 课程包详情 → 消课/新增
用户端：微信授权 → 绑定学员（店长分享卡片）→ 课时首页 → 记录/提醒
```

**关键页面：**
- 店长端：学员列表页（搜索+筛选）、课程包卡片页、消课确认弹窗
- 用户端：课时总览页（大数字展示）、消课时间线页、到期提醒页

#### P5 代码Prompt库（重写）

**组件清单替换：**
1. `LoginPage` — 店长登录（账号密码/微信扫码）
2. `StudentList` — 学员列表（搜索+筛选+新增按钮）
3. `CoursePackageCard` — 课程包卡片（总课时/剩余/进度条/有效期）
4. `CheckInModal` — 消课确认弹窗（课时选择+确认+撤销）
5. `TimelinePage` — 消课记录时间线（用户端）
6. `DataDashboard` — 数据看板（剩余课时统计/到期预警列表）
7. `api/student` `api/checkin` `api/package` — 后端API

#### P6 设计原型（重写）

**核心页面原型：**
- 店长端首页：今日待消课学员列表（快速签到入口）
- 学员详情页：课程包横向卡片 + 消课记录时间线
- 用户端首页：剩余课时大字 + 有效期倒计时 + 最近消课记录

## 四、场景二：台球线下约球 + 球房评价小程序

### 业务本质
O2O服务链路。核心是"人找球房→预约时段→到场履约→评价反馈"。

### 不适用测评框架的部分
- 同样不需要"题库"和"画像体系"
- 但"海报裂变"非常适用（球房打卡海报、战绩海报）

### 适配后的Phase体系

#### P2 功能体系（替代原"内容体系"）

**服务链路拆解：**

| 链路阶段 | 功能点 | 涉及角色 |
|----------|--------|---------|
| 发现 | 附近球房列表、地图模式、搜索筛选 | 玩家 |
| 浏览 | 球房详情（照片/价格/设施/实时空闲台数） | 玩家 |
| 预约 | 选日期→选时段→选台号→确认预约 | 玩家+球房 |
| 履约 | 到店扫码签到、计时开始、提前结束 | 玩家+球房 |
| 支付 | 按小时计费/按局计费、优惠券抵扣 | 玩家 |
| 评价 | 球房评分（环境/设施/服务）、带图评价 | 玩家 |
| 社交 | 发布约球帖（时间/球房/人数/水平要求） | 玩家 |
| 响应 | 报名约球、私信沟通、加微信 | 玩家 |

**P2 产出物：**
- `服务链路功能清单.md`
- `数据字典.md`（球房/台桌/预约单/评价/约球帖表结构）

#### P3 产品方案（重写）

**核心流程：**
```
玩家端：首页推荐 → 球房详情 → 预约时段 → 确认到店 → 扫码开台 → 结束计费 → 评价
约球社交：发布约球帖 → 浏览帖子 → 报名 → 私信 → 加微信
球房端（如有）：预约管理 → 台桌状态 → 营收统计
```

**关键页面：**
- 首页：附近球房feed + 顶部约球帖子流
- 球房详情页：轮播图 + 价格 + 实时台数 + 设施标签 + 评价摘要
- 预约页：日历选择器 + 时段网格（已约满变灰）+ 台号选择
- 约球帖卡片：头像/时间/球房/人数/水平要求/已报名人数

#### P5 代码Prompt库（重写）

**组件清单替换：**
1. `VenueFeed` — 球房列表（卡片式/地图式切换）
2. `VenueDetail` — 球房详情页（轮播+价格+设施+评价）
3. `BookingCalendar` — 预约日历+时段选择器
4. `MatchPostCard` — 约球帖卡片
5. `CheckInQR` — 到店扫码签到
6. `ReviewForm` — 评价表单（星级+标签+文字+图片）
7. `api/venue` `api/booking` `api/match` — 后端API

#### P6 设计原型（重写）

**核心页面原型：**
- 球房卡片（缩略图+距离+价格/小时+实时空闲台数）
- 时段选择网格（绿色=可约/灰色=已满/橙色=选中）
- 约球帖流（时间+球房+已报名人数+水平要求标签）

## 五、两个项目的MVP裁剪建议

### 教培消课 MVP（2-3周）

**必须保留：**
- 店长端：学员新增 + 课程包创建 + 单次消课 + 剩余课时查询
- 用户端：课时总览 + 消课记录（通过店长分享卡片绑定）
- 数据：学员表 + 课程包表 + 消课记录表

**可裁剪：**
- 批量消课（先做单次）
- 财务报表（先做简单统计）
- 续费支付（个人主体做不了，先跳转联系店长）
- 多门店管理（先做单店）
- 消息推送（先不做，用微信自带提醒）

### 台球约球 MVP（3-4周）

**必须保留：**
- 球房列表 + 球房详情（照片/价格/地址）
- 预约时段选择 + 预约确认
- 到店扫码签到（生成临时二维码）
- 基础评价（星级+文字）

**可裁剪：**
- 实时台数（先做静态，后期接球房端手动更新）
- 约球社交（先做球房预约，社交第二期）
- 在线支付（个人主体做不了，先预约后到店付）
- 优惠券系统（第二期）
- 地图模式（先做列表模式）
