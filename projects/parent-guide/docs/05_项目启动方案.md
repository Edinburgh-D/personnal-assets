### 3.2 画像计算引擎（纯前端）

```typescript
// utils/calculator.ts
interface AssessmentResult {
  primaryPortrait: string;      // 主画像ID
  mixPortrait?: string;         // 混合型ID（如有）
  dimensions: {
    A: number;  // 情绪回应（温室园丁）
    B: number;  // 规则锚定（目标教练）
    C: number;  // 探索追问（侦探型）
    D: number;  // 行动果断（指挥官）
  };
  answers: string[];  // 用户选择的选项ID
}

export function calculatePortrait(answers: string[]): AssessmentResult {
  const counts = { A: 0, B: 0, C: 0, D: 0 };
  answers.forEach(ans => counts[ans as keyof typeof counts]++);

  const total = answers.length;
  const dimensions = {
    A: Math.round((counts.A / total) * 100),
    B: Math.round((counts.B / total) * 100),
    C: Math.round((counts.C / total) * 100),
    D: Math.round((counts.D / total) * 100),
  };

  // 找最大值
  const entries = Object.entries(dimensions);
  entries.sort((a, b) => b[1] - a[1]);

  const primary = entries[0][0];
  const secondary = entries[1][0];

  // 判断是否有混合型（前两名差距<15%）
  const isMixed = entries[0][1] - entries[1][1] < 15;

  return {
    primaryPortrait: portraitMap[primary],
    mixPortrait: isMixed ? mixMap[`${primary}+${secondary}`] : undefined,
    dimensions,
    answers,
  };
}
```

### 3.3 菱形风筝图（Canvas前端绘制）

```typescript
// components/KiteChart/KiteChart.tsx
import { Canvas } from '@tarojs/components';

interface KiteChartProps {
  dimensions: {
    A: number;  // 情绪回应（上）
    B: number;  // 规则锚定（右）
    C: number;  // 探索追问（下）
    D: number;  // 行动果断（左）
  };
  primaryColor: string;
}

// Canvas绘制逻辑（Taro兼容版）
function drawKite(ctx, dimensions, width, height) {
  const centerX = width / 2;
  const centerY = height / 2;
  const maxRadius = Math.min(width, height) * 0.4;

  // 4个顶点坐标
  const top = { x: centerX, y: centerY - maxRadius * (dimensions.A / 100) };
  const right = { x: centerX + maxRadius * (dimensions.B / 100), y: centerY };
  const bottom = { x: centerX, y: centerY + maxRadius * (dimensions.C / 100) };
  const left = { x: centerX - maxRadius * (dimensions.D / 100), y: centerY };

  // 绘制四边形
  ctx.beginPath();
  ctx.moveTo(top.x, top.y);
  ctx.lineTo(right.x, right.y);
  ctx.lineTo(bottom.x, bottom.y);
  ctx.lineTo(left.x, left.y);
  ctx.closePath();
  ctx.fillStyle = 'rgba(255, 140, 66, 0.2)';  // 主色半透明
  ctx.fill();
  ctx.strokeStyle = '#FF8C42';
  ctx.stroke();
}
```

### 4.1 核心接口

```typescript
// app/api/assessment/submit/route.ts
import { NextRequest, NextResponse } from 'next/server';

export async function POST(req: NextRequest) {
  const { ageRange, answers, userId } = await req.json();

  // 1. 计算画像（复用前端相同算法）
  const result = calculatePortrait(answers);

  // 2. 保存到数据库
  await db.assessment.create({
    data: {
      userId,
      ageRange,
      answers,
      result,
      createdAt: new Date(),
    },
  });

  // 3. 返回结果
  return NextResponse.json({
    success: true,
    data: {
      assessmentId: uuid(),
      result,
      // 免费层：只给主画像+1句描述
      freeLayer: {
        portraitName: result.primaryPortrait,
        description: getBriefDescription(result.primaryPortrait),
      },
      // 付费层提示
      paidLayer: {
        preview: "完整报告包含：9画像对比 + 逐题解析 + 专属成长路径",
        price: 990,  // 9.9元，单位分
      },
    },
  });
}
```

## 七、关键技术决策

### 7.1 为什么选Taro而不是原生微信小程序？

| 维度 | 原生小程序 | Taro |
|------|-----------|------|
| 多端复用 | ❌ 只能跑微信 | ✅ 微信+H5+RN+百度/支付宝 |
| 开发体验 | WXML+WXSS+JS | React+TypeScript，生态成熟 |
| 组件生态 | 有限 | 复用React生态，npm包丰富 |
| 团队学习成本 | 需学新语法 | 已有React经验可直接上手 |
| 性能 | 原生最优 | 接近原生，Taro3运行时优化后差距小 |

**结论**：团队有React经验 → 选Taro。未来扩展H5/APP无额外成本。

### 7.2 为什么选Next.js而不是Express/Nest.js？

| 维度 | Express | Nest.js | Next.js |
|------|---------|---------|---------|
| 全栈能力 | 纯后端 | 纯后端 | ✅ React + API同仓库 |
| 部署便利性 | 需配Nginx | 需配Nginx | ✅ Vercel一键部署 |
| 服务端渲染 | ❌ | ❌ | ✅ SSR/SSG支持 |
| 团队学习成本 | 低 | 中 | 低（已有React经验） |
| TypeScript支持 | 需额外配置 | ✅ 内置 | ✅ 内置 |

**结论**：Next.js API Routes足够承载MVP后端，未来业务复杂了可平滑迁移到独立服务。

### 7.3 画像计算放前端还是后端？

| 方案 | 前端计算 | 后端计算 |
|------|---------|---------|
| 性能 | ✅ 即时响应，无网络延迟 | ⚠️ 需等API返回 |
| 安全性 | ⚠️ 规则暴露，可篡改 | ✅ 规则隐藏 |
| 灵活性 | ⚠️ 更新需发版 | ✅ 随时调整算法 |
| MVP适用 | ✅ 推荐 | ❌ 过度设计 |

**决策**：MVP阶段画像计算纯前端做（算法简单，就是计数+排序）。后端只存结果不计算，减少API依赖。V2.0如需AI生成个性化文案，再迁移到后端。

### 7.4 题库放前端JSON还是后端数据库？

| 方案 | 前端JSON | 后端数据库 |
|------|---------|-----------|
| 首屏加载 | ✅ 无额外请求 | ⚠️ 需请求题库API |
| 更新灵活性 | ⚠️ 需发版 | ✅ 随时更新 |
| MVP适用 | ✅ 推荐 | ❌ 过度设计 |

**决策**：MVP阶段题库放前端JSON（24道题体积<100KB）。后端只存测评记录。V2.0如需动态题库（每日一练、UGC题目），再迁移到数据库。

| 风险 | 概率 | 影响 | 应对 |
