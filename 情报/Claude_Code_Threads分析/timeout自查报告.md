## 二、根因分析（3大类）

### 1. 消息/内容过大导致处理超时
**占比：** ~40%  
**表现：** 一次输出3000-5000字，或携带图片base64，触发引擎输出限制或超时  
**例子：**
- 西安房价分析初稿（5000+字）直接发群里 → 被截断
- 手机端截图（webp文件）尝试发群里 → timeout

**根因：** 没有预估输出量，没有设置断点分批发。

### 2. 多操作串行导致累积延迟
**占比：** ~35%  
**表现：** 一个任务里同时执行「分析→写文件→推GitHub→汇报」，任何一步卡住导致整体超时  
**例子：**
- 分析帖子→写threads_claude_code_analysis.md→推parent-test→汇报 → 推GitHub时网络延迟拖垮整体

**根因：** 缺乏"保存进度+异步收尾"意识，把所有操作绑在一个事务里。

### 3. 网络/认证外部因素
**占比：** ~25%  
**表现：** GitHub push失败、API调用超时、飞书消息发送超时  
**例子：**
- 推GitHub时"src refspec main does not match any" → 分支名错误导致重试
- 手机端推GitHub"终端没配置GitHub token"

**根因：** 环境状态未预检（remote/branch/token是否就绪）。

## 三、优化措施（已执行+待执行）

### 已执行
- ✅ 分析文件写到本地先保存，再推GitHub（分离操作）
- ✅ 推GitHub改用/tmp干净克隆，避免workspace分支冲突
- ✅ 长内容先发摘要到群里，完整文件推GitHub链接

### 待执行（立即生效）
1. **内容分批发** — 超过2000字的回复，先发"上半部分→保存→下半部分"，或用文件替代
2. **操作三步法** — (1)写文件保存 → (2)推GitHub（独立命令） → (3)汇报结果（摘要）
3. **环境预检** — 任何外部操作（推GitHub、发飞书消息）前，先检查remote/token/权限
4. **降级策略** — 推GitHub失败时，自动降级为"文件内容截图/文本粘贴到群里"

| 因素 | 之前（易timeout） | 本次（无timeout） |

| 任务拆分 | 单一大任务 | 分析文件写好后，推GitHub独立执行 |

**结论：** "先沉淀，后汇报"模式比"边做边发"更稳定。

## 五、给手机端的建议（跨端协同）

手机端timeout更频繁，原因不同：
- **截图发送** — 图片太大，改用文字描述+关键信息摘录
- **浏览卡顿** — Threads滚动加载慢，分批浏览（20条/次）而非一次性100条
- **消息风暴** — 一次发太多条消息到群里，改为"批量收集→整理后一次性发汇总"

**建议分工：**
- 手机端：负责浏览+复制原始内容（轻量操作）
- 电脑端：负责分析+写文件+推GitHub（重计算操作）
